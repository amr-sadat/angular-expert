## Overview

Angular's i18n system is built around `@angular/localize`. It provides two ways to deliver translations:

1. **Build-time locale baking** — the Angular CLI compiles a separate application bundle per locale. Each bundle has translations inlined at compile time. Zero runtime overhead; requires deploying one build artifact per locale.
2. **Runtime translation loading** — a single bundle ships with message IDs; `loadTranslations()` injects translations at startup before the app bootstraps. Suitable when locales are selected at login or stored in user preferences.

| Mechanism | Bundle count | Runtime cost | Dynamic locale switch |
|---|---|---|---|
| Build-time baking (`--localize`) | One per locale | None | No — requires page reload or redeploy |
| Runtime loading (`loadTranslations`) | One | Small lookup cost | No — must reload to re-apply |

Both mechanisms use the same `i18n` attribute and `$localize` API for marking messages. The extraction and translation workflow is identical.

## Setup: ng add @angular/localize

```bash
ng add @angular/localize
```

This command:
- Adds `@angular/localize` to `package.json` dependencies.
- Adds `"@angular/localize/init"` to the `types` array in `tsconfig.app.json`, making `$localize` available globally in TypeScript without an import.
- Does **not** configure locales in `angular.json` — you do that manually (see [build-time baking](#build-time-locale-baking) below).

After running the command, `$localize` is available in any `.ts` file without an explicit import.

## $localize Tagged Template Literals

`$localize` marks strings in TypeScript code for extraction and translation. It uses the tagged template literal syntax.

### Basic usage

```typescript
import { Component, ChangeDetectionStrategy } from '@angular/core';

@Component({
  selector: 'app-header',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<h1>{{ title }}</h1>`,
})
export class HeaderComponent {
  title = $localize`Welcome to the app`;
}
```

### With interpolations

Variable placeholders are preserved in the extracted message. Translators see a placeholder token; the runtime substitutes the runtime value.

```typescript
@Component({
  selector: 'app-greeting',
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<p>{{ greeting }}</p>`,
})
export class GreetingComponent {
  name = input.required<string>();

  greeting = computed(() => $localize`Hello, ${this.name()}:name:`);
}
```

The `:name:` suffix is a placeholder name — it appears in the extracted XLIFF file, helping translators understand what the interpolation represents.

### Meaning, description, and custom ID

The metadata block comes before the message text, separated by a colon. It is optional but strongly recommended for messages that could be ambiguous.

```typescript
// Syntax: $localize`:meaning|description@@customId:message text`

// Description only
const label = $localize`:A button label for submitting the login form:Sign in`;

// Meaning + description (meaning disambiguates identical source strings)
const headerTitle = $localize`:site header|The main page heading:Welcome`;

// Custom ID (stable across source-text changes)
const navItem = $localize`:@@nav.home:Home`;

// All three
const fullExample = $localize`:site header|An introduction header@@introductionHeader:Hello i18n!`;
```

**Why custom IDs matter:** Without a custom ID, Angular generates a message ID from the source text. Editing the source string changes the ID and orphans existing translations. Assign a stable custom ID to every string that is expected to evolve.

## i18n Attribute in Templates

Mark element text content for extraction by adding the `i18n` attribute.

### Basic text

```html
<h1 i18n>Hello i18n!</h1>
```

### With meaning, description, and custom ID

The attribute value follows the same `meaning|description@@id` syntax as `$localize`:

```html
<!-- Description only -->
<p i18n="The introductory paragraph on the home page">Welcome to our platform.</p>

<!-- Meaning + description + custom ID -->
<h1 i18n="site header|The main page heading@@pageTitle">Angular App</h1>

<!-- Custom ID only -->
<button i18n="@@submitButton">Submit</button>
```

### Marking attributes

Use `i18n-attributeName` to mark an element attribute for translation:

```html
<!-- Mark the title attribute -->
<img [src]="logoSrc" i18n-title title="Angular logo" alt="Angular logo" />

<!-- Mark aria-label -->
<button i18n-aria-label aria-label="Close dialog" (click)="close()">×</button>
```

### Elements with no DOM output

Use `<ng-container>` to group translatable text without adding an element to the DOM:

```html
<ng-container i18n>This text has no wrapper element in the DOM.</ng-container>
```

### Dynamic interpolations in templates

Interpolations inside an `i18n`-marked element are included in the extracted message as numbered placeholders:

```html
<p i18n="@@lastUpdated">Last updated: {{ lastUpdated() }}</p>
```

The extractor outputs the placeholder as `<x id="INTERPOLATION"/>` in XLIFF. Translators move the placeholder to the correct position in their locale's sentence structure.

## Message Extraction: ng extract-i18n

Run extraction after marking all strings. The CLI scans templates and TypeScript files for `i18n` attributes and `$localize` tags.

### Default (XLIFF 1.2)

```bash
ng extract-i18n
```

Produces `src/locale/messages.xlf` by default.

### Specifying format and output path

```bash
# XLIFF 1.2 (default) — widely supported by CAT tools (Phrase, Lokalise, Weblate)
ng extract-i18n --format=xlf --output-path=src/locale

# XLIFF 2 — newer format, more expressive, not all tools support it yet
ng extract-i18n --format=xlf2 --output-path=src/locale

# XMB — Google's format, required for Google Translate Toolkit
ng extract-i18n --format=xmb --output-path=src/locale

# JSON — simple key/value, easiest to work with programmatically
ng extract-i18n --format=json --output-path=src/locale

# ARB (Application Resource Bundle) — Flutter/Dart ecosystem standard
ng extract-i18n --format=arb --output-path=src/locale
```

### Workflow

1. Run `ng extract-i18n` to produce `messages.xlf` (the source file).
2. Copy `messages.xlf` to `messages.fr.xlf`, `messages.de.xlf`, etc.
3. Send the locale-specific files to translators (or a translation platform).
4. Receive translated files back and place them in `src/locale/`.
5. Wire them into `angular.json` (build-time) or load them via `loadTranslations()` (runtime).

Re-run extraction whenever new strings are added. Existing translations in locale files are preserved — only new message IDs are added as untranslated entries.

## Runtime Translation Loading vs Build-Time Locale Baking

### Build-time locale baking

Configure locales in `angular.json`:

```json
{
  "projects": {
    "my-app": {
      "i18n": {
        "sourceLocale": "en-US",
        "locales": {
          "fr": {
            "translation": "src/locale/messages.fr.xlf"
          },
          "de": {
            "translation": "src/locale/messages.de.xlf"
          }
        }
      },
      "architect": {
        "build": {
          "options": {
            "localize": true
          }
        }
      }
    }
  }
}
```

```bash
# Build all configured locales
ng build --localize

# Build a single locale
ng build --localize=fr
```

The CLI outputs one sub-directory per locale: `dist/my-app/fr/`, `dist/my-app/de/`. Your web server must serve the correct sub-directory based on the request (HTTP `Accept-Language` header, URL prefix, or cookie).

**Tradeoffs:**
- Pro: Translations are inlined — no runtime parsing, no async loading delay.
- Pro: Each bundle only contains one locale's strings — smaller than a multi-locale runtime bundle.
- Con: One full build per locale. Large locale sets multiply CI build time.
- Con: Adding a locale requires a new deployment.

### Runtime translation loading

Call `loadTranslations()` **before** `bootstrapApplication()`. Angular processes `$localize` tagged strings the first time they are evaluated; any call to `loadTranslations()` after that point has no effect on already-rendered strings.

```typescript
// main.ts
import '@angular/localize/init';
import { loadTranslations } from '@angular/localize';
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { appConfig } from './app/app.config';

async function prepareApp() {
  const locale = detectUserLocale(); // e.g. read from cookie or navigator.language

  if (locale !== 'en-US') {
    const translations = await fetchTranslations(locale);
    loadTranslations(translations);
  }

  await bootstrapApplication(AppComponent, appConfig);
}

prepareApp();

function detectUserLocale(): string {
  return localStorage.getItem('locale') ?? navigator.language ?? 'en-US';
}

async function fetchTranslations(locale: string): Promise<Record<string, string>> {
  const response = await fetch(`/assets/i18n/${locale}.json`);
  return response.json();
}
```

`loadTranslations` expects a `Record<string, string>` mapping message IDs to translated strings. For XLIFF files, parse them server-side or with a build step that converts them to JSON before deployment.

**Tradeoffs:**
- Pro: One build artifact for all locales.
- Pro: Adding a locale requires only a new JSON file — no rebuild or redeployment.
- Con: Translation loading adds a network round-trip before bootstrap. Mitigate with preloading or inline the most common locale.
- Con: Language switching requires a full page reload — `loadTranslations` does not re-translate already-rendered strings.
- Con: The runtime translation parser adds a small amount of code to the bundle (~10 KB gzipped).

## SSR and i18n: withI18nSupport()

By default, Angular skips hydration for components that contain i18n blocks. Without `withI18nSupport()`, the client cannot reconcile translated text nodes produced by the server with the expected DOM structure, causing a hydration mismatch or silent dehydration of the affected subtree.

Add `withI18nSupport()` to `provideClientHydration()`:

```typescript
// app.config.ts
import { ApplicationConfig } from '@angular/core';
import {
  provideClientHydration,
  withI18nSupport,
  withEventReplay,
  withIncrementalHydration,
} from '@angular/platform-browser';
import { provideRouter } from '@angular/router';
import { provideZonelessChangeDetection } from '@angular/core';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideZonelessChangeDetection(),
    provideRouter(routes),
    provideClientHydration(
      withI18nSupport(),
      withEventReplay(),
      withIncrementalHydration(),
    ),
  ],
};
```

**What `withI18nSupport()` does:** It enables Angular's hydration engine to match translated text nodes on the client to their server-rendered counterparts, even when the text content differs between locales. Without it, any component with an `i18n`-marked element will be fully re-rendered on the client (dehydrated), discarding the server-rendered DOM and causing a flash of unstyled or missing content.

> For the full SSR and hydration provider configuration, see [references/ssr-and-performance.md](references/ssr-and-performance.md).

### Build-time i18n with SSR

When using build-time locale baking with SSR, each locale gets its own server bundle. Configure `@angular/ssr` alongside `--localize`:

```bash
ng build --localize
```

Angular generates separate server entry points per locale. Your reverse proxy must route requests to the correct locale's server bundle based on the URL prefix or `Accept-Language` header.

## ICU Message Formats

ICU (International Components for Unicode) message syntax handles pluralization and gender/selection without `@if` branching in templates.

### Plural

```html
<span i18n>
  {itemCount, plural,
    =0 {No items}
    =1 {One item}
    other {{{ itemCount }} items}
  }
</span>
```

### Select (gender or enumeration)

```html
<span i18n>The author is {gender, select, male {male} female {female} other {other}}</span>
```

### Nested ICU (plural inside select)

```html
<span i18n>
  Updated:
  {minutes, plural,
    =0 {just now}
    =1 {one minute ago}
    other {{{ minutes }} minutes ago by {gender, select, male {male} female {female} other {other}}}
  }
</span>
```

ICU expressions are most commonly used inside `i18n`-marked template elements, where the extractor preserves the full ICU structure in the output catalog.

## Anti-patterns

### Calling loadTranslations() after bootstrapApplication

```typescript
// WRONG — translations loaded after bootstrap have no effect on already-rendered strings
bootstrapApplication(AppComponent, appConfig).then(() => {
  loadTranslations(myTranslations); // too late
});

// CORRECT — load translations before bootstrapping
loadTranslations(myTranslations);
bootstrapApplication(AppComponent, appConfig);
```

### Omitting withI18nSupport() in SSR apps

```typescript
// WRONG — hydration silently skips i18n components; translated subtrees are re-rendered client-side
provideClientHydration(withEventReplay())

// CORRECT
provideClientHydration(withI18nSupport(), withEventReplay())
```

This is the most common SSR + i18n mistake. The symptom is a content flash or console warnings about hydration mismatches on any component with `i18n` attributes.

### Building i18n strings at runtime without $localize

```typescript
// WRONG — the extractor cannot find this string; it will never be translated
const label = 'Hello, ' + userName;

// CORRECT — $localize captures the structure; the placeholder is preserved in XLIFF
const label = $localize`Hello, ${userName}:userName:`;
```

### Using i18n attribute on elements with Angular control flow

The `i18n` attribute marks only the **direct text content** of the element it is on. Wrapping an `@if` block with a single `i18n` attribute does not translate the content inside the conditional branch:

```html
<!-- WRONG — the i18n attribute is on the outer div but the text is inside @if -->
<div i18n>
  @if (loggedIn()) {
    <span>Welcome back!</span>
  }
</div>

<!-- CORRECT — put i18n directly on the element containing the text -->
<div>
  @if (loggedIn()) {
    <span i18n>Welcome back!</span>
  }
</div>
```

### Missing custom IDs on evolving strings

```html
<!-- FRAGILE — editing the source text silently orphans all translations -->
<h1 i18n>Welcome to our new homepage</h1>

<!-- STABLE — custom ID survives source text edits -->
<h1 i18n="@@pageTitle">Welcome to our new homepage</h1>
```

Assign a stable `@@id` to any message that is likely to be copy-edited over time.
